# KuluoBishi (推箱子)

基于 Unreal Engine 5.5 的推箱子游戏，内置关卡编辑器，支持自定义关卡的创建与游玩。

## 试玩说明

### 基本操作

- **WASD / 方向键** — 控制角色在网格上移动（上下左右四方向）
- 角色面前有箱子时，会自动推动箱子前进
- **撤销** — 走错了可以撤回上一步
- **暂停菜单** — 可重新开始或返回关卡选择

### 游戏目标

玩家抵达目标点（Goal）即可通关。

### 关卡机制

| 地块类型 | 说明 |
|---------|------|
| **地板（Floor）** | 普通可通行地块 |
| **墙壁（Wall）** | 不可通行的障碍 |
| **目标点（Goal）** | 箱子的目的地，所有目标被占满即通关 |
| **压力板（Pressure Plate）** | 箱子放上后触发机关，可联动开启门 |
| **门（Door）** | 默认关闭阻挡通行，由压力板组控制开关 |
| **冰面（Ice）** | 角色和箱子会在冰面上持续滑行，直到碰到障碍物停下 |

### 教程系统

预设关卡支持配置教程提示，引导玩家逐步掌握操作和机制：

- 教程以文字弹窗形式出现在屏幕上，按步骤推进
- 每一步可配置**触发条件**（何时显示）和**完成条件**（何时消失），两者使用同一套条件类型：
  - `Immediate` — 立即触发（关卡开始或上一步完成后）
  - `OnPlayerMove` — 玩家成功移动
  - `OnPushBox` — 玩家推动箱子
  - `OnDoorOpened` — 门被打开
  - `OnGridPosition` — 玩家到达指定格子
  - `AfterSteps` — 玩家走了 N 步
  - `OnUndo` / `OnReset` — 玩家执行撤销或重置
  - `None` — 仅用于完成条件，表示手动点击按钮推进
- 教程数据通过 DataAsset 按关卡索引配置，无教程的关卡不会显示任何提示
- 暂停菜单打开时教程弹窗自动隐藏，关闭后恢复

#### 编辑器教程

关卡编辑器同样支持教程引导，帮助新用户快速上手编辑器操作：

- 复用与游玩教程相同的弹窗和流程机制
- 通过 `OnGameplayEvent` 条件类型 + EventTag 字符串匹配编辑器操作，无需新增枚举值
- 可用的 EventTag：
  - `BrushChanged` — 切换笔刷
  - `CellPainted` — 放置格子
  - `CellErased` — 擦除格子
  - `GroupCreated` — 创建分组
  - `ModeChanged` — 编辑模式切换（如进入压力板放置模式）
  - `NewLevel` — 新建关卡
  - `LevelSaved` — 保存关卡
  - `LevelLoaded` — 加载关卡
  - `LevelTested` — 测试关卡
- 编辑器教程在 DataAsset 的 `EditorTutorialSteps` 数组中配置，与游玩教程独立

### 关卡模式

- **预设关卡** — 游戏自带的关卡，按顺序解锁
- **自定义关卡** — 玩家通过编辑器创建的关卡

## 关卡编辑器说明

从主菜单进入关卡编辑器，可以自由设计推箱子关卡。

### 工具栏

| 操作 | 说明 |
|------|------|
| **新建（New）** | 创建指定尺寸的空白关卡 |
| **保存（Save）** | 将当前关卡保存为 JSON 文件 |
| **加载（Load）** | 加载已有的自定义关卡 |
| **测试（Test）** | 直接进入游玩模式测试当前关卡 |
| **返回（Back）** | 返回主菜单 |

### 笔刷系统

侧边栏提供 9 种笔刷，左键绘制，右键擦除：

- **Floor** — 铺设地板
- **Wall** — 放置墙壁
- **Ice** — 铺设冰面
- **Goal** — 放置目标点
- **Door** — 放置门（需要关联压力板组）
- **Pressure Plate** — 放置压力板
- **Box Spawn** — 设置箱子初始位置
- **Player Start** — 设置玩家出生点（唯一）
- **Eraser** — 擦除地块

### 分组机制

门和压力板通过"组"进行关联：

1. 放置门后，进入放置压力板模式，为该门关联压力板
2. 同一组内的所有压力板被触发后，对应的门将打开
3. 通过 **Group Manager** 面板管理所有分组，支持创建、删除和自定义颜色
4. 每个组可配置颜色，通过 **GroupColorIndicatorComponent**（继承自 StaticMeshComponent）在编辑器和游戏中实时显示组颜色。直接在机关蓝图中添加该组件、设置 Mesh 和材质即可使用；也支持蓝图子类重载 `OnUpdateVisual` 自定义视觉效果

### 关卡验证

保存或测试时会自动验证关卡的合法性，检查包括：
- 是否存在玩家出生点
- 箱子与目标点数量是否匹配
- 门是否正确关联了压力板组
- 所有元素是否可达

## 设计思路

推箱子看起来简单，但实际开发中需要回答几个核心问题：**人怎么在格子上移动？机制之间如何互不干扰地协作？玩家走错了怎么办？怎么让编辑器和游玩共享同一套逻辑？**

### 人怎么动？—— 网格离散化

推箱子的特点是角色按格子移动，而不是自由走动。如果用物理碰撞来做，对齐网格会非常痛苦。

我的做法是将所有位置抽象为整数坐标 `FIntPoint`，由一个 `GridManager` 统一管理网格状态和占位关系。角色想移动时，只需向 GridManager 询问"目标格子能不能走"，GridManager 根据格子类型、是否有障碍物、门是否打开等条件做判断，将复杂的空间逻辑简化为整数坐标运算。移动、推箱子、滑行全部基于同一套网格查询完成。

### 箱子什么时候动？机关之间怎么配合？—— 组件化机制 + 事件驱动

地图上有多种机制（目标点、压力板、门），它们的触发条件和效果各不相同。如果写在一起，每加一个新机制都要改一大片代码。

我的思路是把每种机制做成独立的组件，继承自 `GridMechanismComponent`，挂载到网格上的 `TileVisualActor`。压力板负责检测占位，门负责开关通行，目标点只做胜负标记——各司其职，互不知道对方的存在。它们之间通过 Unreal 多播委托通信（如 `OnActorLogicalMoved`、`OnPitFilled`），GridManager 只管广播事件，不关心谁在监听。这样新增机制只需写一个新组件，不需要改动已有代码。

压力板和门通过"组"进行关联：同一组的所有压力板都被触发后，对应的门才会打开。这个 AND 逻辑足以覆盖经典推箱子的关卡设计需求，同时保持规则简单可预测。

### 冰面滑行怎么做？—— Modifier 与 Mechanism 的分离

冰面的行为和门、压力板有本质区别：它不是一个有状态的"机关"，而是一个**修改移动行为的被动属性**——踩上去就继续滑，仅此而已。

如果把冰面也塞进 `GridMechanismComponent`，就意味着它要背上激活/去激活、分组、阻挡通行等与自身无关的接口。所以我另外设计了 `TileModifierComponent` 体系：它只需实现一个 `ShouldContinueMovement()` 方法，告诉 GridManager "这个格子上的角色/箱子应该继续沿当前方向移动"。GridManager 在移动逻辑中检查目标格子是否有 Modifier，如果有就调用 `CalculateIceSlideDestination()` 计算最终落点，将整段滑行作为一次原子移动处理。

这样 Mechanism 负责"有状态的交互"（门开关、压力板触发），Modifier 负责"无状态的行为修饰"（冰面滑行），两套体系各自独立，职责清晰。未来如果要加传送带、弹簧地板之类的新机制，只需判断它属于哪一类，选择对应的基类继承即可。

### 走错了怎么办？—— 快照栈撤销

推箱子很容易走进死局，撤销是刚需。常见方案是记录每一步的"操作"然后逆向回放，但推箱子的状态联动复杂（推箱子可能连锁触发压力板、开关门），逆向回放需要处理大量边界情况。

我选择了更直接的方案：每次移动前，在 GameState 中压入一份完整的关卡快照（`FLevelSnapshot`），记录玩家位置、所有箱子位置、门的开关状态和步数。撤销时直接弹出上一份快照恢复整个场景。实现简单可靠，天然支持任意步数回退，也不需要担心联动状态的一致性问题。

### 教程怎么不侵入游戏逻辑？—— WorldSubsystem + 统一条件模型

教程系统需要感知玩家操作（移动、推箱子、开门等），但如果把教程逻辑嵌入 GameMode 或 Character，会让核心代码变得臃肿且难以关闭。

我的做法是将教程管理器实现为 `UWorldSubsystem`（`TutorialSubsystem`），它独立于 GameMode 生命周期，不持有任何游戏逻辑，只通过 `NotifyCondition()` 等轻量通知接口接收外部事件。GameMode、Character、PushableBox、DoorMechanism 各自在关键时刻调用一行通知代码，教程系统据此判断是否推进——通知方对教程的存在完全无感。

触发条件和完成条件共用同一套 `FTutorialCondition` 结构体和 `ETutorialConditionType` 枚举，这意味着"玩家到达某格子"既可以作为显示教程的触发条件，也可以作为关闭教程的完成条件，不需要维护两套独立的条件系统。教程内容通过 DataAsset 配置，关卡设计者可以在编辑器中直接编辑每个关卡的教程步骤，无需修改任何代码。

这套设计的通用性在编辑器适配中得到了验证：关卡编辑器通过已有的 `OnGameplayEvent` + `EventTag` 机制发送编辑器操作通知（如笔刷切换、放置格子、保存关卡等），无需新增任何枚举值或修改条件匹配逻辑，只需在 `LevelEditorGameMode` 中添加一行 `NotifyGameplayEvent` 调用即可接入。为避免编辑器场景下教程干扰光标和输入模式，`TutorialSubsystem` 通过 `bIsEditorTutorial` 标记跳过输入模式切换。

### 编辑器和游玩怎么共存？—— 分离 GameMode，共享数据

编辑器需要画笔、验证、分组管理，游玩需要移动、推箱子、胜负判定——两者的交互逻辑完全不同，但操作的都是同一张网格。

我的做法是让编辑器和游玩使用各自独立的 GameMode（`LevelEditorGameMode` / `SokobanGameMode`），但共享同一个 `GridManager` 和同一套 JSON 数据格式。编辑器的"测试"功能直接将当前网格序列化为 `FLevelData`，切换到游玩 GameMode 加载，实现无缝衔接。预设关卡和自定义关卡走完全相同的加载流程，编辑器产出的关卡可以直接被游戏读取，没有任何格式转换。
