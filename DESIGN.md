# 技术策划文档

## 项目概述

基于 Unreal Engine 5.5 的推箱子解谜游戏 Demo。玩家在封闭房间内推动箱子触发机关、打开门，最终到达出口通关。项目包含完整的运行时关卡编辑器，策划可以在游戏内直接设计、验证和测试关卡，无需程序介入。

### 项目亮点

- **运行时关卡编辑器**：笔刷绘制 + 分组管理 + 自动验证 + 即时测试，策划可独立完成关卡全流程
- **组件化机制系统**：压力板-门联动、冰面滑行、传送阵等扩展机制，新增机关只需添加组件 + 填描述表，不改已有代码
- **事件驱动架构**：中央事件总线解耦所有子系统，教程、UI、音效等消费方即插即用
- **描述表驱动扩展**：地块类型和编辑器笔刷均通过描述表集中定义，单行配置即可贯通序列化、渲染、编辑器和运动校验

## 玩法设计

### 核心玩法

经典推箱子规则：玩家在网格上四方向移动，面前有箱子时自动推动。将所有箱子推到目标点（Goal）即可通关。

### 扩展机制

在基础推箱子之上，设计了三种扩展机制来增加关卡设计空间：

| 机制 | 类型 | 行为 | 设计意图 |
|------|------|------|----------|
| **压力板 + 门** | 有状态交互 | 箱子压住压力板触发分组，同组所有压力板激活后门打开 | 引入"钥匙-锁"关系，创造多步解谜 |
| **冰面** | 无状态修饰 | 角色和箱子在冰面上持续滑行直到碰壁 | 改变移动规则，增加空间规划难度 |
| **传送阵** | 有状态交互 | 成对放置，支持双向/单向传送 | 打破空间连续性，扩大关卡设计可能性 |

#### 为什么把机制分为"有状态交互"和"无状态修饰"两类？

压力板、门、传送阵都有明确的状态变化（开/关、激活/未激活），需要分组管理和事件通信。而冰面只是被动地修改移动行为——踩上去就继续滑，没有"激活"的概念。

如果把所有机制塞进同一个基类，冰面就要背上激活/去激活、分组、阻挡通行等与自身无关的接口。所以设计了两套独立体系：

- **Mechanism（有状态）**：继承 `GridMechanismComponent`，支持激活/去激活、分组、通行控制
- **Modifier（无状态）**：继承 `TileModifierComponent`，只需实现 `ShouldContinueMovement()` 一个方法

两者各自独立，职责清晰。未来加传送带、弹簧地板等新机制时，只需判断它属于哪一类，选择对应基类继承即可。

### 撤销系统

推箱子很容易走进死局，撤销是刚需。常见方案是记录每一步操作然后逆向回放，但推箱子的状态联动复杂（推箱子连锁触发压力板、开关门），逆向回放需要处理大量边界情况。

采用更直接的方案：每次移动前压入一份完整的关卡快照（玩家位置、所有箱子位置、门开关状态、步数），撤销时直接弹出恢复。实现简单可靠，天然支持任意步数回退，不需要担心联动状态的一致性问题。

## 关卡编辑器设计

### 设计目标

让策划和玩家能够**独立完成关卡设计的完整流程**：构思 → 搭建 → 验证 → 测试 → 保存，全部在游戏内闭环完成，不需要程序支持，不需要切换工具。

### 笔刷系统

编辑器提供 10 种笔刷，覆盖所有地块类型：

| 笔刷 | 快捷键 | 说明 |
|------|--------|------|
| 地板 Floor | 1 | 铺设可通行地面 |
| 墙壁 Wall | 2 | 放置不可通行障碍 |
| 冰面 Ice | 3 | 铺设滑行地面 |
| 目标点 Goal | 4 | 放置箱子目标位置 |
| 门 Door | 5 | 放置门，自动进入压力板放置流程 |
| 压力板 Pressure Plate | 6 | 放置压力板，归入当前选中组 |
| 箱子 Box Spawn | 7 | 设置箱子初始位置 |
| 玩家起点 Player Start | 8 | 设置玩家出生点（唯一） |
| 传送阵 Teleporter | 9 | 成对放置传送点 |
| 橡皮擦 Eraser | 0 | 擦除地块 |

**交互设计考量**：

- **左键绘制，右键擦除**：最少操作步骤，策划不需要频繁切换笔刷和橡皮擦
- **数字键快捷切换**：10 种笔刷对应 1-0 键，单手可完成所有切换
- **上下文感知的放置流程**：放置门后自动进入压力板放置模式，放置传送阵自动进入配对模式，引导策划完成机关的完整配置而不会漏配

### 分组机制

门和压力板的关联关系对策划来说不够直观——"这个压力板控制哪扇门？"是编辑关卡时最常见的困惑。设计方案：

1. **颜色可视化**：每个分组分配颜色，同组的门和压力板在编辑器和游戏中显示相同颜色，一眼可辨
2. **Group Manager 面板**：集中管理所有分组，支持创建、删除、重命名和自定义颜色
3. **引导式放置**：放置门时自动创建新组并进入压力板放置模式，确保门总有对应的触发器

传送阵复用同一套分组系统，两个传送点通过相同的 GroupId 配对，并支持切换传送方向（双向/单向入口/单向出口）。

### 关卡验证

策划会犯错，好的工具应该主动拦截错误而不是让策划在测试时才发现问题。编辑器在保存和测试前自动执行验证：

| 检查项 | 级别 | 说明 |
|--------|------|------|
| 玩家出生点 | 错误 | 必须存在且位于有效格子上 |
| 目标点 | 错误 | 至少存在一个目标点 |
| 门的压力板配置 | 警告 | 门所在组是否有对应的压力板 |
| 传送阵配对 | 警告 | 同组传送阵是否恰好为 2 个 |
| 箱子数量 | 警告 | 是否放置了箱子 |

错误会阻止保存/测试，警告仅提示但不阻断。这样既防止了明显的配置错误，又不会过度限制策划的创作自由度。

### 编辑器与游玩的无缝衔接

编辑器的"测试"功能实现了零延迟的编辑-测试循环：

1. 点击"测试"→ 验证通过 → 将当前网格序列化为 `FLevelData`
2. 切换到游玩 GameMode → 加载刚才序列化的数据 → 直接开始游玩
3. 退出测试 → 回到编辑器，继续编辑

编辑器和游玩使用各自独立的 GameMode，但共享同一个 GridManager 和同一套 JSON 数据格式。预设关卡和自定义关卡走完全相同的加载流程。

### 编辑器教程

编辑器本身也支持教程引导，帮助新用户快速上手操作。复用与游玩教程相同的弹窗和流程机制，通过事件总线广播编辑器操作事件（笔刷切换、格子绘制、保存加载等），教程系统自动匹配条件推进步骤。不需要为编辑器教程编写任何额外代码。

## 教程系统设计

### 设计目标

教程需要感知玩家操作（移动、推箱子、开门等），但不能侵入游戏核心逻辑。

### 方案：事件总线 + 统一条件模型

游戏使用中央事件总线 `GameEventBus`（`UWorldSubsystem`），所有游戏逻辑事件通过标签统一广播。

教程管理器 `TutorialSubsystem` 订阅关心的事件，收到后自动进行条件匹配。它不持有任何游戏逻辑，也不被任何游戏类直接引用。

触发条件和完成条件共用同一套条件结构体，"玩家到达某格子"既可以作为显示触发，也可以作为关闭条件，不需要维护两套系统。

教程内容通过 DataAsset 配置，关卡设计者在编辑器中直接编辑每个关卡的教程步骤，无需修改代码。

### 支持的条件类型

| 条件 | 说明 |
|------|------|
| `Immediate` | 立即触发 |
| `OnPlayerMove` | 玩家移动 |
| `OnPushBox` | 推动箱子 |
| `OnDoorOpened` | 门被打开 |
| `OnGridPosition` | 到达指定格子 |
| `AfterSteps` | 走了 N 步 |
| `OnUndo` / `OnReset` | 撤销或重置 |
| `OnGameplayEvent` | 通用事件标签匹配（编辑器教程使用） |
| `None` | 手动点击推进 |

### 通用性验证

编辑器教程复用了完全相同的教程系统。编辑器通过事件总线广播 `Editor.*` 标签事件（如 `Editor.BrushChanged`、`Editor.CellPainted`），教程系统用 `OnGameplayEvent` 条件匹配，无需新增任何枚举值或修改匹配逻辑。这验证了事件总线架构的可扩展性。

## 扩展性设计

### 新增一个地块类型需要几步？

以"尖刺地板"为例（踩上去会受伤的地块）：

**第 1 步：添加枚举值**（GridTypes.h）
```cpp
enum class EGridCellType : uint8 {
    // ...existing...
    Spike,
};
```

**第 2 步：填描述表**（GridTypes.cpp）
```cpp
{ EGridCellType::Spike, "Spike", true, true, true },
//                       序列化名  可通行  需要地板底层  擦除变地板
```

这一行配置自动贯通：JSON 序列化/反序列化、GridManager 通行校验、编辑器擦除行为、视觉渲染层级。

**第 3 步：写机制组件**
```cpp
class USpikeMechanismComponent : public UGridMechanismComponent {
    virtual void OnActivate() override;   // 尖刺弹出
    virtual bool BlocksPassage() const override { return false; }
};
```

**第 4 步：添加编辑器笔刷**（EditorBrushTypes.cpp）
```cpp
{ EEditorBrush::Spike, EGridCellType::Spike, TEXT("尖刺"), TEXT("Spike"), TEXT("S"), FLinearColor::Red },
```

**第 5 步：创建蓝图并注册到 Catalog**

继承 `TileActor` 创建蓝图，放入想要的 Mesh 和材质，挂上第 3 步写的机制组件。然后打开 `DA_DefaultTileStyles`，添加一条 `FTileVisualStyle`：
- `StyleId`：唯一标识（如 `SpikeDefault`）
- `ApplicableType`：适用的地块类型（如 `Spike`）
- `ActorClass`：刚做的蓝图

完成。不需要修改 GridManager、LevelSerializer、编辑器绘制逻辑或 UI 侧边栏的任何已有代码。注册后系统自动处理：编辑器侧边栏出现选择按钮、缩略图自动生成、关卡序列化自动保存加载、类型校验自动防止错误搭配。

### 给已有地块类型新增视觉变体

同一种地块类型可以有多种视觉样式（比如地板有木纹、石砖、草地等变体），通过 `TileStyleCatalog`（DataAsset）集中管理。只需重复上面的第 5 步：继承 `TileActor` 做一个新蓝图，然后在 `DA_DefaultTileStyles` 里加一行。美术或策划可以独立完成，不需要程序改任何代码。

### 新增一个 Modifier 需要几步？

以"传送带"为例（踩上去沿固定方向移动）：

**第 1 步**：同上，添加枚举值和描述表行

**第 2 步**：继承 `TileModifierComponent`，重写一个方法
```cpp
class UConveyorModifier : public UTileModifierComponent {
    virtual bool ShouldContinueMovement(EMoveDirection Direction) const override {
        return Direction == ConveyorDirection;  // 只在传送带方向上继续移动
    }
};
```

GridManager 在移动逻辑中自动检查 Modifier 并应用效果，无需修改移动代码。

## 关卡设计

### 5 关预设关卡的设计思路

关卡按照"基础教学 → 机制引入 → 综合挑战"的节奏编排：

| 关卡 | 尺寸 | 引入机制 | 箱子数 | 设计意图 |
|------|------|----------|--------|----------|
| Level 1 | 8×5 | 基础推箱子 | 0 | 熟悉移动和普通目标点，最简单的入门 |
| Level 2 | 8×9 | 压力板 + 门 | 2 | 引入"推箱子到压力板开门"的核心循环，引入需要开门的目标点 |
| Level 3 | 8×8 | 冰面 + 传送阵 | 1 | 改变移动规则 |
| Level 4 | 9×10 | 多组门序列 | 2 | 多分组门的顺序解锁，需要规划推箱子的先后顺序 |
| Level 5 | 9×10 | 全机制综合 | 3 | 3 个箱子 + 3 个分组，要求同时管理多条解谜线路 |

每一关只引入一个新概念，让玩家在理解上一个机制后再面对新挑战。关卡 4 和 5 不引入新机制，而是通过增加箱子数和分组数来提升组合复杂度。
